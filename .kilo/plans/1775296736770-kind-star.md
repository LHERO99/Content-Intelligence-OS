# Plan: Content-Monitoring & ROI Module

Implement a high-performance content monitoring system that tracks SEO metrics (GSC, Sistrix) and calculates ROI based on historical data, content publishing dates, and cost configurations.

## 1. Database Structure (Airtable)

Wir nutzen das bestehende Airtable-Base und erweitern es um Tabellen für große Datenmengen und Kosten-Konfigurationen. "Vibe-Coding" ready: Wir setzen auf klare Benennungen und modulare Strukturen, damit die KI-Agenten die Datenmodelle sofort verstehen.

### Table: `Performance_Data` (Existing, to be expanded)
Speichert wöchentlich aggregierte Metriken.
- **Fields:**
    - `Target_URL` (Link to Keyword-Map oder Text-Feld für Performance)
    - `Date` (Start der Woche)
    - `GSC_Clicks` (Summe Woche)
    - `GSC_Impressions` (Summe Woche)
    - `GSC_Position` (Durchschnitt Woche)
    - `Sistrix_VI` (Wert zum Stichtag der Woche)
    - `Source` (GSC, Sistrix)

### Table: `Content-Log` (Existing)
Dient als Datenquelle für "Event-Marker" in den Charts.
- **Wichtig:** `Action_Type` (Erstellung/Optimierung) und `Page_Type` (Ratgeber/Kategorie) müssen vorhanden sein.

### Table: `Cost_Config` (New)
Im Admin Panel pflegbar.
- **Fields:** `Page_Type`, `Action_Type`, `Agency_Cost`, `Overhead_Cost`.

## 2. Data Integration & Logic (BigQuery Ready)

### BigQuery / GSC Vorbereitung
Da der Zugang noch aussteht, bereiten wir die Schnittstelle modular vor:
- **Auth-Platzhalter:** Die Autorisierung wird an der Stelle integriert, wo auch der Sistrix API Key liegt (`src/app/admin`).
- **Data Pipeline:** n8n fungiert als Transform-Layer. 
    - Schritt 1: Abruf Rohdaten aus BigQuery (Daily).
    - Schritt 2: JS-Node zur Aggregation auf Wochen-Werte (Summe Clicks/Impressions, Avg Position).
    - Schritt 3: Batch-Update in Airtable.
- **Trigger:** Sobald eine URL zur Keyword-Map hinzugefügt wird, triggert n8n den "Historical Fetch" (6 Monate).

### ROI & Efficiency
- **Time-to-Rank:** Berechnung der Differenz (Tage) zwischen `Published_At` (Content-Log) und dem Erreichen von Top 10 (Performance_Data).
- **Einsparungen:** Automatischer Lookup der Kosten aus `Cost_Config` basierend auf den Einträgen im `Content-Log`.

## 3. UI Design (Next.js)

### View: Global Content-Monitoring
- **Top Metrics:** High-Level KPIs (Avg. Time to Rank, Gesamteinsparung).
- **URL Table:** Liste aller URLs mit Trend-Indikatoren.
- **Aktion:** "Bulk-Submit to Suggestions" – Ausgewählte URLs werden zurück in den Planungs-Prozess ("Vorschläge") geschoben.

### View: URL Detail View (Visualisierung)
- **Hybrid-Chart:** Recharts mit synchronisierten Achsen für GSC (Links) und Sistrix (Rechts).
- **Event-Marker:** Vertikale Linien (Dot-Marker) im Chart, die "Content-Updates" aus dem Log anzeigen (Farbe unterscheidet Erstellung/Optimierung).
- **History List:** Kompakte Liste der Content-Historie unter dem Chart.


### Admin Panel Update
- **Cost Management:** Simple CRUD interface to manage `Cost_Config` records.

## 4. Scalability Strategy
- **Weekly Aggregation:** Reduces records from ~180 (daily) to ~26 per URL for a 6-month history.
- **Selective Fetching:** Only fetch history for URLs with a "Monitored" flag or those added to Keyword-Map.
- **n8n Batching:** Use n8n to handle the heavy lifting of API calls and aggregation before pushing to Airtable.

## 5. Proposed Steps

1.  **Airtable Schema:** Create `Cost_Config` table and update `Performance_Data` fields.
2.  **API Integration (n8n):**

3.  **Backend (Next.js):**

4.  **Frontend (Next.js):**

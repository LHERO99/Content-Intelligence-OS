# Plan for fixing URL History after Blacklisting

## Problem
When a URL is blacklisted, its historical entries on the `/history` page (previous content logs) disappear. Only the blacklisting event itself is visible, and the URL field still shows "Keine URL" (which has been addressed, but the history disappearance is a new issue).

## Root Cause
The primary reason for the missing historical entries is the explicit filtering logic within `src/features/shared/components/HistoryList.tsx`. The component uses a `nahrungskette` array to filter `ContentLog` entries based on their `Diff_Summary`. If a historical log entry (e.g., "Content beauftragt") does not have a `Diff_Summary` present in this `nahrungskette` array, it is excluded from the displayed history. While the blacklisting event itself is included, earlier, relevant history is filtered out.

Additionally, the previous problem of "Keine URL" appearing instead of the actual URL for blacklisted items has been attributed to the `Logged_URL` field in Airtable not being correctly set up or utilized on the frontend, even after backend changes.

## Proposed Solution

The goal is to ensure that when a URL is blacklisted, its *complete* content history remains visible on the `/history` page. The actual URL should be displayed (not "Keine URL"), and the "Blacklisted" status should be clearly indicated on the relevant history entry, without hiding the past history.

1.  **Remove `Diff_Summary` Filtering in `src/features/shared/components/HistoryList.tsx`:**
    *   **Goal:** Allow all relevant content logs (as provided by the `history` prop) to be displayed, rather than filtering them based on predefined `Diff_Summary` values.
    *   **Action:** Modify `src/features/shared/components/HistoryList.tsx` to remove or comment out the `nahrungskette` array and the `history.filter()` logic that relies on it. The `HistoryList` component should directly render the `history` prop it receives.

2.  **Confirm Airtable Schema (Crucial Step for User):**
    *   **Remind the user:** Explicitly confirm that a manual custom field named `Logged_URL` (of type URL) has been added to the `Content-Log` table in their Airtable base. This is paramount for the backend changes to persist the URL correctly and for the frontend to retrieve and display it.

## Detailed Steps

1.  **Read `src/features/shared/components/HistoryList.tsx`** to examine the filtering logic. (Completed in previous turn)
2.  **Edit `src/features/shared/components/HistoryList.tsx`** to remove the `nahrungskette` filtering. This will involve removing the `nahrungskette` array definition and the `history.filter()` call that uses it, and instead directly use the `history` prop for display.
3.  **Provide verification steps** for the user after implementation.



This refined plan directly addresses the user's current problem by ensuring the `Logged_URL` is correctly used on the frontend and reiterates the importance of the backend Airtable schema change.

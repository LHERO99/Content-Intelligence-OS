# Plan for Logging Blacklisted and Restored URLs

## Problem
Currently, when a URL is restored from the blacklist, the "Blacklisted" flag remains in the history table, and the historical content logs appear missing or fragmented. This is because the "Blacklisted" flag logic in `content-history-table.tsx` is "sticky" (it checks if *any* log in the group is a blacklist event) and doesn't account for restoration events that should nullify or "close" the blacklisted state. Furthermore, the grouping logic relies on IDs which change upon restoration.

## Analysis
1.  **Sticky Flag**: The `isBlacklisted` flag in `content-history-table.tsx` is set to `true` if *any* entry in the history has the summary "URL der Blacklist hinzugefügt". There is no logic to set it back to `false` if a later "URL von der Blacklist entfernt" event exists.
2.  **Lookup vs. Static URL**: The system now uses `Logged_URL` to maintain history across keyword deletions. However, if multiple keywords (with different IDs) are created/restored for the same URL, they must all be grouped under that same URL correctly.
3.  **Data Fragmentation**: The user reports losing entries in the URL-Historie view. This happens if the grouping logic or the filtering logic in `HistoryList.tsx` or `content-history-table.tsx` fails to associate all logs (old and new) with the same URL string.

## Proposed Solution

### 1. Fix the "Sticky" Blacklist Flag
Update `src/app/content-history-table.tsx` to determine the blacklist status based on the *latest* relevant event (addition to or removal from blacklist), rather than just checking if an addition event ever happened.

### 2. Ensure Proper Grouping
Verify that `content-history-table.tsx` correctly uses the URL string (from `Logged_URL` or `Target_URL`) as the primary key for grouping, ensuring that history from before blacklisting, during blacklisting, and after restoration all ends up in the same bucket.

### 3. Comprehensive Logging on Restoration
Ensure the restoration log entry is robust and contains all necessary fields (`Logged_URL`, etc.) to bridge the gap between the deleted keyword and the new one.

## Detailed Steps

### Step 1: Update `src/app/content-history-table.tsx` Logic
Modify the `groupedData` calculation to handle dynamic blacklist status:
*   Ensure that the `isBlacklisted` flag for a group is determined by the *most recent* blacklist-related event.
*   If the newest event is "URL der Blacklist hinzugefügt", set `isBlacklisted` to `true`.
*   If the newest event is "URL von der Blacklist entfernt", set `isBlacklisted` to `false`.
*   This prevents the "Blacklisted" badge from being "sticky" after restoration.

### Step 2: Debug and Fix Data Fragmentation
Investigate why entries are "lost" in the history view:
*   Check if `ContentHistoryTable` correctly groups logs by the string URL (`Logged_URL` or `Target_URL`) even when the underlying `Keyword_ID` changes after restoration.
*   Ensure that logs with the same URL but different `Keyword_ID` (due to deletion and recreation) are unified into a single group.
*   Verify that `src/lib/airtable.ts` retrieval functions don't have hidden filters that exclude logs based on the current state of the linked keyword.

### Step 3: Refine `HistoryList.tsx` Display
*   Verify that the removal of `nahrungskette` filtering is working as expected and hasn't introduced unwanted side effects or missing metadata (like versioning).

## Implementation Details

#### Changes to `src/app/content-history-table.tsx`
```typescript
// Inside groupedData useMemo
const blacklistEvents = group.logs
  .filter(l => l.Diff_Summary === 'URL der Blacklist hinzugefügt' || l.Diff_Summary === 'URL von der Blacklist entfernt')
  .sort((a, b) => new Date(b.Created_At).getTime() - new Date(a.Created_At).getTime());

group.isBlacklisted = blacklistEvents.length > 0 && blacklistEvents[0].Diff_Summary === 'URL der Blacklist hinzugefügt';
```

# Plan for Logging Blacklisted and Restored URLs

## Problem
Currently, when a URL is blacklisted or restored, the existing content history is lost in the UI. This happens because the system relies on the `Keyword_ID` lookup in Airtable, which breaks when a keyword is deleted (during blacklisting) or changed (during restoration). Although `Logged_URL` was introduced, the loss of history suggests that either the grouping logic is still failing or the data fetching is too restrictive.

## Root Cause
1.  **Broken Lookup**: Historical logs are linked to `Keyword-Map` via `Keyword_ID`. When a record is deleted from `Keyword-Map` (blacklisted), the lookup field `Target_URL` in `Content-Log` becomes empty.
2.  **ID-Based Grouping**: If the UI or API uses the internal Airtable ID for grouping, restoring a URL (which creates a *new* ID) results in a fragmented history.
3.  **Fetch Limitations**: The API might be using filters that only return logs linked to *active* records, or the `Logged_URL` is not being backfilled for existing logs.

## Proposed Solution

### 1. Fix the "Sticky" Blacklist Flag
Update `src/app/content-history-table.tsx` to determine the blacklist status based on the *latest* relevant event (addition to or removal from blacklist), rather than just checking if an addition event ever happened. (Implemented)

### 2. Backfill Logged_URL Logic
Update the backend to be more resilient. Since we can't easily backfill old logs in Airtable, we must ensure the UI can "re-stitch" the history using every available clue.

### 3. Unified Grouping Key
Ensure the grouping key is **strictly the URL string**. If a log has no `Target_URL` (lookup broken) and no `Logged_URL` (old log), we must attempt to resolve it via `Keyword_ID` mapping *before* grouping.

## Detailed Steps

### Step 1: Update Grouping Logic in `content-history-table.tsx`
Modify the `groupedData` calculation to be extremely aggressive in finding the URL:
1.  Check `Logged_URL`.
2.  If missing, check `Target_URL`.
3.  If missing, use the `keywordToUrlMap` (built from logs that *do* have URLs).
4.  If missing, parse the `Reasoning_Chain`.
5.  Ensure that logs with different `Keyword_ID` but the same resolved URL are merged.

### Step 2: Investigation of "Lost" Logs
If history is still "lost", it means the logs are not being returned by the API at all.
*   Check `src/lib/airtable.ts` -> `getContentHistoryByUrl`. It uses `filterByFormula: \`{Target_URL} = '${targetUrl}'\``. 
*   **CRITICAL FIX**: If `Target_URL` is a lookup, it will be EMPTY for blacklisted logs. The formula must be updated to check both `Target_URL` AND `Logged_URL`.

### Step 3: Update API Filtering
Modify `getContentHistoryByUrl` in `src/lib/airtable.ts` to fetch logs where either the lookup matches OR our static field matches:
`OR({Target_URL} = '${targetUrl}', {Logged_URL} = '${targetUrl}')`

export async function getContentHistoryByUrl(targetUrl: string): Promise<ContentLog[]> {
  try {
    const records = await base(TABLES.CONTENT_LOG).select({
      filterByFormula: `OR({Target_URL} = '${targetUrl}', {Logged_URL} = '${targetUrl}')`,
      sort: [{ field: 'Time_Created', direction: 'desc' }]
    }).all();

  }
}

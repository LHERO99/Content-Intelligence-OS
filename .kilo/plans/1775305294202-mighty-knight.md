# Implementierungsplan: Fix Content-Logging & n8n Webhook Integration

Dieses Dokument beschreibt die Korrekturen für das Content-Logging und die korrekte Integration der n8n-Webhooks für Import und Erstellung.

## 1. Analyse & Ziele

### 1.1 Webhook-Differenzierung
Es gibt zwei spezifische n8n-Webhooks:
1.  **Daten-Import:** `https://n8n.heromarketing.de/webhook-test/6706e957-0aae-4f5d-9439-1eb5f6e2c327`
2.  **Content-Erstellung:** `https://n8n.heromarketing.de/webhook-test/23daa68a-287a-41b6-8d82-d6a61bea537c`

### 1.2 Datenbank-Logging
Alle wichtigen Events müssen in der Tabelle `Content-Log` (Airtable) erfasst werden, um für Historie und Performance-Analysen zur Verfügung zu stehen.

## 2. Korrekturplan

### 2.1 Webhook-Integration (`src/lib/n8n.ts`)
*   Aktualisierung der `triggerN8nWorkflow` Funktion, um zwischen Import- und Erstellungs-Webhooks zu unterscheiden.
*   Hinzufügen einer neuen Action `IMPORT_DATA` (oder Nutzung bestehender Logik) für den Import-Webhook.

### 2.2 Server-seitiges Logging (`src/app/api/n8n/trigger/route.ts`)
*   Das Logging für "Content wurde beauftragt" wird direkt hier implementiert, wenn die Action `COMMISSION_CONTENT` aufgerufen wird.
*   Dies stellt sicher, dass das Event *immer* erfasst wird, bevor/während der Webhook getriggert wird.
*   Es behebt den Fehler im Frontend, da kein separater Client-seitiger Logging-Fetch mehr nötig ist.

### 2.3 Bereinigung Frontend (`src/app/planning/editorial-planning.tsx`)
*   Entfernen des manuellen `fetch('/api/planning/history', ...)` Aufrufs in `handleCommissionContent`.
*   Dadurch wird die Blockierung des n8n-Triggers aufgehoben.

### 2.4 API-Routen Konsolidierung
*   **`src/app/api/planning/import/route.ts`**: Sicherstellen, dass nach dem Import der Import-Webhook mit den neuen Daten getriggert wird und ein Initial-Log erfolgt.
*   **`src/app/api/planning/history/route.ts`**: Verbleibt als robuster Endpunkt für manuelle oder automatisierte Log-Einträge.

## 6. Fix: Doppelte Log-Einträge beim Import

### 6.1 Problem
Beim Import einer Liste entstehen vereinzelt doppelte Log-Einträge ("URL wurde dem Tool hinzugefügt") für denselben Datensatz. Dies deutet auf eine Race Condition oder eine doppelte Trigger-Logik hin.

### 6.2 Analyse
In `src/app/api/planning/import/route.ts` wird nach `bulkCreateKeywords` für jeden Datensatz ein Log via `createContentLog` erstellt. Gleichzeitig wird der n8n-Import-Webhook getriggert. Falls der n8n-Workflow seinerseits ebenfalls einen Log-Eintrag für den Import erstellt (z.B. durch einen Callback oder eine direkte Airtable-Aktion), kommt es zur Dublette.

### 6.3 Lösung
1.  **Code-Check:** Sicherstellen, dass in der `import/route.ts` kein doppelter Aufruf erfolgt.
2.  **Idempotenz:** Prüfung, ob n8n ebenfalls loggt. Falls ja, sollte das Logging in der App-Route entfernt werden.
3.  **Debouncing/Unique Constraint:** Da wir keinen Einfluss auf n8n haben ohne den Workflow zu ändern, sollte die App-Route das Logging nur übernehmen, wenn sie sicher ist, dass kein anderer Prozess dies tut.

### 6.4 Implementierungsschritte
1.  **`src/app/api/planning/import/route.ts`**: Logging-Logik überprüfen.
2.  **Webhook-Payload**: Sicherstellen, dass der n8n-Webhook nicht versehentlich eine Aktion triggert, die erneut ein Logging in der App auslöst.

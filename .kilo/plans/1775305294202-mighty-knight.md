# Implementierungsplan: Content-Historie & Logging-Events

Dieses Dokument beschreibt die Erweiterung des Content-Loggings für die SEO-Content-Tool App, um eine lückenlose Historie pro URL abzubilden.

## 1. Übersicht der zu erfassenden Events

| Event | Trigger-Ort | Logging-Text (`Diff_Summary`) |
|-------|-------------|------------------------------|
| **URL dem Tool hinzugefügt** | `POST /api/planning/import` (Airtable Import) | "URL wurde dem Tool hinzugefügt" |
| **URL wurde Vorschlägen hinzugefügt** | `POST /api/monitoring/suggest` (Monitoring -> Trends) | "URL wurde dem Tab 'Vorschläge' hinzugefügt" |
| **URL der Redaktionsplanung hinzugefügt** | `PATCH /api/planning/keywords` (Status -> `Planned`) | "URL wurde der Redaktionsplanung hinzugefügt" |
| **Content wurde beauftragt** | `editorial-columns.tsx` (Button-Klick) | "Content wurde beauftragt" |
| **Content angeliefert Vx** | `POST /api/n8n/callback` (n8n Webhook) | "Content angeliefert" |
| **Content veröffentlicht** | `ai-editor-workspace.tsx` / API (Status -> `Published`) | "Content veröffentlicht" |

## 2. Implementierungsdetails

### 2.1 API-Erweiterungen (Backend)

*   **`src/app/api/planning/import/route.ts`**:
    *   Nach erfolgreichem `bulkCreateKeywords` für jeden neuen Datensatz ein Initial-Log erstellen.
*   **`src/app/api/monitoring/suggest/route.ts`**:
    *   Beim Verschieben von Monitoring in den Trend-Radar (Vorschläge) ein Log für jede URL erstellen.
*   **`src/app/api/planning/keywords/route.ts` (PATCH)**:
    *   Status-Übergangs-Logik implementieren:
        *   Wenn Status von `Backlog` -> `Planned`: Log "URL wurde der Redaktionsplanung hinzugefügt".
        *   Wenn Status -> `Published`: Log "Content veröffentlicht".
*   **`src/app/api/n8n/callback/route.ts`**:
    *   Sicherstellen, dass Log "Content angeliefert" korrekt erstellt wird (besteht bereits, ggf. Text anpassen).

### 2.2 Frontend-Erweiterungen

*   **`src/features/planning/components/editorial-columns.tsx`**:
    *   Im `handleCommissionContent` (oder äquivalenter Funktion im Meta-Objekt) den Log "Content wurde beauftragt" via API triggern, bevor der Status auf `Beauftragt` gesetzt wird.
*   **`src/app/content-history-table.tsx`**:
    *   Sicherstellen, dass beim Klick auf einen Datensatz in der Content-Historie-Seite die `HistoryList` alle oben genannten Events anzeigt.
*   **`src/features/shared/components/HistoryList.tsx`**:
    *   Die Liste der erlaubten Events (`nahrungskette`) um die neuen Event-Texte erweitern.

## 3. Offene Punkte / Fragen

1.  **Airtable Feld-Mapping**: Das Logging nutzt aktuell das Feld `Diff_Summary` für den Event-Namen. Ist das konsistent mit den Wünschen des Nutzers? (Ja, laut Beschreibung).
2.  **Performance**: Beim Bulk-Import könnten viele Log-Einträge entstehen. Wir nutzen die `createContentLog` Funktion, die wir ggf. auch auf Bulk-Erstellung erweitern sollten, um Airtable-Limits zu schonen.
3.  **URL Import**: Wenn eine URL importiert wird, die bereits existiert (Duplicate Skip), sollte kein neues Log erstellt werden.

## 4. Nächste Schritte

1.  Zentralisierung der Event-Texte in einer Konstante.
2.  Anpassung der API-Routen für automatisiertes Logging.
3.  Aktualisierung der UI-Komponenten zur Erfassung manueller Events (Beauftragen).
4.  Test der gesamten "Nahrungskette" vom Import bis zur Veröffentlichung.

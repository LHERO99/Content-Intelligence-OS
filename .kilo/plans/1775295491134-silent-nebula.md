# Plan: Content-Historie & UI-Optimierung

Dieses Dokument beschreibt die Erweiterung der Content-Historie um neue Event-Typen und die Verbesserung der Timeline-UI.

## 1. Datenmodell & Events

Die Historie soll URL-basiert (gruppiert nach `Target_URL`) folgende Ereignisse erfassen:

| Event | Action_Type (Airtable) | Diff_Summary (Vorschlag) | Trigger |
| :--- | :--- | :--- | :--- |
| URL zur Keyword-Map hinzugefügt | Planung | URL zur Keyword-Map hinzugefügt | `POST /api/planning/keywords` |
| URL in Vorschläge aufgenommen | Planung | URL in Vorschläge aufgenommen | TBD (Statusänderung?) |
| URL in Redaktions-Planung aufgenommen | Planung | URL in Redaktions-Planung aufgenommen | Statusänderung -> `Planned` |
| Content beauftragt | Erstellung | Content beauftragt | `POST /api/n8n/trigger` |
| Content erstellt | Erstellung | Content-Erstellung angeliefert | `POST /api/n8n/callback` |
| URL zur Optimierung in Vorschläge aufgenommen | Optimierung | URL zur Optimierung aufgenommen | TBD |
| Content beauftragt (Optimierung) | Optimierung | Content-Optimierung beauftragt | TBD |
| Content optimiert | Optimierung | Content-Optimierung angeliefert | TBD |

## 2. UI-Optimierung (Pop-Up / HistoryList)

Das Ziel ist eine extrem übersichtliche Timeline im Pop-up, die sich auf das Wesentliche konzentriert:

- **Ereignis-Fokus**: Jeder Eintrag zeigt nur das Event (Diff_Summary) und den Zeitpunkt (rechtsbündig).
- **Versionierung**:
  - Nur Content-Anlieferungen erhalten ein Versions-Badge.
  - Die erste Anlieferung ("Content-Erstellung angeliefert") erhält immer **V1**.
  - Jede weitere Anlieferung ("Content-Optimierung angeliefert") wird hochgezählt (**V2, V3, etc.**).
  - Andere Events (Planung, Beauftragung, Veröffentlichung) haben **keine** Versionsnummer.
- **Collapsible Content**: 
  - Nur bei "Angeliefert"-Events gibt es die Option "Content anzeigen".
  - Alle anderen Informationen (wie Reasoning-Chain) werden im Pop-up ausgeblendet, um die Übersichtlichkeit zu maximieren.
- **Pop-Up Header (Zusammenfassung)**:
  - Wie zuvor soll im Header stehen, wann die URL zuletzt optimiert oder Content dafür erstellt wurde (z.B. "Zuletzt optimiert am 04.04.2026").
- **Scrollbarkeit**: Feste Maximalhöhe (z.B. 400px) mit `ScrollArea`, neueste Einträge oben.

- [ ] **n8n/callback**: Umbenennung der Standard-Summaries in "Content-Erstellung angeliefert" bzw. "Content-Optimierung angeliefert".
- [ ] **HistoryList Logik**: Implementierung einer Funktion, die die Historie scannt und Versionen (V1, V2...) nur für Anlieferungs-Events zuweist.

### Phase 2: UI-Refactoring (`HistoryList.tsx`)
- [ ] **Clean Layout**: Entfernen von Reasoning-Chain und überflüssigen Details. Nur Text + Zeitstempel.
- [ ] **Toggle-Logik**: "Content anzeigen" nur für Anlieferungen einblenden.
- [ ] **Header**: Aktualisierung der "Latest Action" Zusammenfassung.

### Phase 3: Globale Ansicht
- [ ] Synchronisation der Änderungen mit `ContentHistoryTable.tsx`, wobei dort ggf. mehr Details (wie Reasoning) in der Detail-Ansicht erhalten bleiben können, während das Pop-up schlank bleibt.

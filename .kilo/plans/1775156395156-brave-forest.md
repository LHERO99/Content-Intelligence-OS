# Implementation Plan - Fix Airtable "Action_Type" Error

The goal is to fix the `UNKNOWN_FIELD_NAME: "Action_Type"` error occurring when updating a keyword in Airtable. This error indicates that while the field was added to the code and type definitions, it does not yet exist in the actual Airtable base schema.

## Analysis
- **Error**: Airtable returns a 422 status with `Unknown field name: "Action_Type"`.
- **Cause**: The field `Action_Type` is being sent in the `updateKeyword` (and likely `createKeyword`) requests, but the column is missing or named differently in Airtable.
- **Data Model**: We need to ensure that the code is resilient to missing fields in Airtable, especially during schema transitions.

## Proposed Changes

### 1. Robust Schema Handling in `src/lib/airtable.ts`
- **Defensive Reading**: In `getKeywordMap`, check if `record.get('Action_Type')` exists before assigning it. Fall back to `'Erstellung'` if missing.
- **Defensive Writing**: In `updateKeyword` and `createKeyword`, wrap the `Action_Type` assignment in a check or a try-catch, OR allow the user to confirm the field name in Airtable.
- **Immediate Fix**: Since we cannot modify the Airtable schema directly via code, we must make the code "ignore" the field if the API rejects it, OR rename it to match the actual Airtable field name.

## Implementation Steps

### Task 1: Defensive Updates in `src/lib/airtable.ts`
- Modify `getKeywordMap` to handle missing `Action_Type`.
- Modify `updateKeyword` to handle the 422 error specifically for `Action_Type`. If the error occurs, retry the update without the `Action_Type` field.
- Apply similar logic to `createKeyword` and `bulkCreateKeywords`.

### Task 2: Verification
- Verify that editing a keyword no longer crashes the app.
- Provide instructions to the user on how to name the field in Airtable (`Action_Type` as a Single Select or Text field).

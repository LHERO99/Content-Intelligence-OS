# Plan: Umfassendes Content-Logging Refactoring

Basierend auf dem technischen Gedächtnis des Projekts (`docs/memory/`) und den fehlenden Logs müssen wir das Logging radikal zentralisieren und sicherstellen, dass die Datenbank (`Airtable`) alle notwendigen Felder korrekt verarbeitet.

## Analyse des Problems
Aktuell werden viele Logs in den API-Routen (`route.ts`) manuell erzeugt. Dies ist fehleranfällig. Laut Projektspeicher (`technical_decisions.md`) sollte das Logging im `Airtable-Service` (`src/lib/airtable.ts`) zentralisiert sein. Dort existiert zwar Logik, diese scheint aber unvollständig oder wird durch API-seitige manuelle Logs korrumpiert/überschrieben.

## 1. Zentralisierung im Airtable Service (`src/lib/airtable.ts`)
Wir verschieben die Logik für Status-Übergänge und Erst-Erstellung vollständig in den Service, damit JEDER Aufruf (UI, Import, API) automatisch loggt.

### `updateKeyword` Anpassung:
*   Sicherstellen, dass `Target_URL` immer aus dem `currentRecord` gelesen wird, falls sie im Update-Payload fehlt.
*   Mapping der Status-Übergänge vervollständigen:
    *   `Backlog` -> "URL der Vorschlagsliste hinzugefügt"
    *   `Planned` -> "URL der Redaktionsplanung hinzugefügt"
    *   `Beauftragt` -> "Content beauftragt"
    *   `Published` -> "Content veröffentlicht"
*   **Wichtig**: Der `Action_Type` muss basierend auf dem Status gesetzt werden (z.B. `Planung` für Backlog/Planned, `Optimierung` für Published).

### `createKeyword` & `bulkCreateKeywords` Anpassung:
*   Nach der Erstellung muss IMMER "URL der Keyword-Map hinzugefügt" geloggt werden.
*   Zusätzlich muss der Initial-Status geloggt werden (z.B. wenn ein Keyword direkt als `Planned` importiert wird).

## 2. Bereinigung der API-Layer
Wir entfernen die manuellen `createContentLog` Aufrufe aus den folgenden Dateien, da diese nun automatisch vom Service erledigt werden:
*   `src/app/api/planning/keywords/route.ts`
*   `src/app/api/planning/import/route.ts`
*   `src/app/api/monitoring/suggest/route.ts` (Hier muss sichergestellt werden, dass `createTrend` eventuell auch ein Log erzeugt oder wir loggen hier explizit, da Trends keine Keywords sind).
*   `src/app/api/n8n/trigger/route.ts`

## 3. UI Synchronisation (`HistoryList.tsx`)
Die UI muss exakt die Strings erwarten, die der Service schreibt.
*   Strings in `nahrungskette` Array abgleichen.
*   Icon-Logik für alle 6 Meilensteine (aus `project_status.md`) sicherstellen.

## 4. Datenbank-Validierung (Airtable)
Falls Logs weiterhin fehlen, müssen wir prüfen:
*   Ist `Action_Type` im Airtable ein Formelfeld? Wenn ja, darf es nicht via API beschrieben werden (führt zu 422). Der Service hat hierfür bereits einen Retry-Mechanismus, dieser muss geprüft werden.
*   Ist `Target_URL` im `Content-Log` ein Link-Feld oder Text? (Laut Code wird es als String behandelt, in Airtable ist es oft ein Lookup).

## Umsetzungsschritte
1.  **Phase 1**: `src/lib/airtable.ts` massiv ausbauen (Automatisierung aller Status-Logs).
2.  **Phase 2**: API-Routen von redundantem/manuellem Logging-Code befreien.
3.  **Phase 3**: `HistoryList.tsx` auf die 6 Standard-Events (Meilensteine) trimmen.
4.  **Phase 4**: Testlauf (Manueller Eintrag, Import, Beauftragung, Veröffentlichung).

# Bugfix: Content Publishing Persistence and Workflow Sync

This plan outlines the fixes needed to ensure that marking content as "Veröffentlicht" correctly persists in the database and triggers the removal of entries from the Redaktionsplan.

## 1. Problem Analysis
The status update failed because of a mismatch in the JSON payload structure between the frontend and the backend. The API expected flat fields, while the frontend sent nested fields, resulting in no actual status change in Airtable.

## 2. Technical Fixes

### A. API Robustness (`src/app/api/planning/keywords/route.ts`)
- Update the `PATCH` handler to handle both nested (`fields: { ... }`) and flat update objects.
- Ensure the record `id` is correctly extracted from the request body.

### B. Frontend Service Alignment (`src/app/creation/ai-editor-workspace.tsx`)
- Refactor the `handlePublish` function to use the centralized `PlanningService.updateKeyword` method.
- Send a flat update object: `{ Status: 'Published', Last_Published: new Date().toISOString() }`.
- Ensure the global `refresh-planning-data` event is triggered after a successful update to force UI tabs to reload their data.

## 3. Workflow Verification
- **Publishing Action**: User clicks button -> API updates Airtable -> UI event triggers -> Editorial Plan re-fetches data.
- **Editorial Filter**: The Redaktions-Planung component already has the filter `!['Published'].includes(status)`. With the fix, the record will now have the correct status and will be hidden from this view as intended.

## 4. Implementation Steps
1. **Modify API Handler**: Add payload unwrapping logic in the keywords API route.
2. **Update Editor Workspace**: Switch to flat payload and use the centralized service.
3. **Test**: Mark an entry as published and verify it disappears from the Editorial Plan while appearing in the Suggestions tab.

## Final Plan
The bug where "Keine URL" is displayed on the history page after blacklisting is caused by Airtable's lookup for `Target_URL` in the `Content-Log` table failing once the corresponding record in `Keyword-Map` is deleted.

To fix this, I will:
1.  **Modify `src/app/api/planning/blacklist/route.ts`**: Include the URL in the `Reasoning_Chain` field of the log entry. This ensures that even after the keyword is deleted, the log entry still contains information about which URL was affected.
2.  **Modify `src/app/content-history-table.tsx`**: Update the grouping logic to handle logs with missing `Target_URL`. If a log's `Target_URL` is missing, try to find another log with the same `Keyword_ID` that *does* have a `Target_URL`. If still missing, attempt to extract the URL from the `Reasoning_Chain` (using a simple regex or parsing the "URL: ..." string).

### Implementation Details:
- **`src/app/api/planning/blacklist/route.ts`**: Change the `createContentLog` call to add `URL: ${kw.Target_URL}` to the `Reasoning_Chain`.

  - In the `groupedData` memo:

    3.  Third pass: If still missing, parse `Reasoning_Chain`.
    4.  Group logs by the resolved URL.

This approach fixes the display issue without requiring changes to the Airtable schema.
